产品战略构想 · 2026.06.23

AI 猎头
聚合平台

不是封闭的猎头系统，而是招聘行业的「Agent 路由器」——
让任意 Agent 一站式搜人、获取数据、留下数据，并在过程中沉淀数据资产。

核心定位

聚合层

统一接入 Boss、猎聘、LinkedIn、内部人才库、飞书等 100+ 数据源，屏蔽底层差异。

Skill/Harness

把复杂流程封装成可复用 skill：需求理解、多源搜索、简历解析、匹配评估、验证、触达。

数据资产

过路数据沉淀为人才图谱与公司画像，形成网络效应和长期壁垒。

今天跑通的流程

已验证可行，但仍有 6-7 步，目标是压缩到 1 步

Step 1

读取客户驾驶舱

Codex + 飞书 CLI，提取嵌入式职位要求

Step 2

验证需求

智谱 AutoClaw 交叉验证结果准确性

Step 3

多源搜人

OpenManus / 内部人才库 / Boss 积极求职者

Step 4

评估 + 触达

Codex 评估优先级，AutoClaw 验证，自动联系

未来目标 打完客户电话 → 自动生成文档 → Agent 自动找人推人

两大 Sourcing 场景

1

新客户快速嘎嘎推人

  • • 扫完库内存量合适的人
  • • 扫完 Boss 上积极看机会的人
  • • 目前需要好几周，AI 可压缩到小时级
战略： 把效率和成本做到极致，赚取窗口期利润
2

冰山下的人

  • • 水面上的人扫完后
  • • 通过转介绍、Mapping、深挖找人
  • • 依赖关系网络、行业 know-how、信任
终局壁垒： 短期 AI 替代不了，是长期护城河

工程架构：三层 Skill/Harness 模型

入口层

飞书 Bot / Web / API / 客户自有 Agent

↓
编排层

Workflow Harness

流程编排、状态机、人机回环、可观测

↓
Skill 层

可复用原子能力

需求理解 多源搜索 简历解析 匹配评估 验证 触达 文档生成
↓
适配层

飞书 CLI / Boss / 智联 / 猎聘 / LinkedIn / 邮件 / 电话

↓
数据层

TalentMatch DB + 知识图谱 + 交互日志

商业模式演进

Phase A

SaaS 工具

按猎头团队/公司收费，养活团队，验证 PMF。

现状：TalentMatch 正在这条路
Phase B

聚合平台 + 数据网络

按调用量/数据量收费，沉淀人才图谱，形成网络效应。

目标：成为招聘行业数据交换层
Phase C

Agent 工厂

把每个场景封装成可售卖的 Agent：新客户 onboarding、电话后文档、mapping。

终局：能力产品化

建议路径：A → B → C

与 TalentMatch 衔接的落地路径

1

Phase 1：流程自动化（2-4 周）

做 `NewClientWorkflow` harness，集成飞书 CLI、AutoClaw 验证、Boss/人才库搜索，输出排序后的候选人列表 + 客户文档。

2

Phase 2：沉淀为 Skill（4-8 周）

把每个步骤拆成独立 skill，定义 skill 接口（输入/输出/配置），建立 skill registry。

3

Phase 3：开放聚合层（8-12 周）

暴露 API，让外部 Agent 调用，按调用量收费，数据网络效应开始形成。

风险与优先级

高风险 · 数据合规

抓取 Boss/LinkedIn 等平台数据可能触发反爬或法律风险。

高风险 · 数据质量

多源聚合后去重、标准化做不好，推荐质量会崩塌。

高风险 · 客户信任

自动联系候选人语气不当，会损害品牌。

中风险 · 技术债

skill 接口设计不好，后期扩展困难。

中风险 · 模型成本

多 Agent 循环调用，token 成本高。

低风险 · 运维

服务可用性、监控、备份需跟上。
