# 2026-06-23 AI 猎头聚合平台产品构想

## 背景

用户（老板）分享了对 TalentMatch / 猎头业务下一步形态的深度思考。核心思路是：**不要做封闭的猎头系统，而是做类似 OpenRouter 的聚合中间件/中转站**，让用户的 Agent 或入口能够一站式地到 100 个地方搜人、获取数据、留下数据。

## 核心产品形态

**"AI 猎头聚合平台" / "招聘 Agent 路由器"**

- 类似 OpenRouter：统一接口，屏蔽底层多个数据源/Agent 的差异
- 用户自己有自己的 Agent 或入口
- 产品作为中转站，接收过路的数据，沉淀为资产
- 表面像"套壳"，但内部把复杂流程封装成 skill / harness，让产品变得极易使用

## 今天跑通的流程（已验证可行）

### 流程 A：职位需求 → 找人 → 评估 → 联系

1. Codex 读取客户驾驶舱（使用飞书 CLI），了解某家公司嵌入式职位的所有要求
2. 同步用智谱 AutoClaw 验证结果
3. 把要求文档发给 OpenManus（openmai），OpenManus 去系统搜人
4. 搜出来的人再扔给 Codex 重新评估（使用人才库 API 识别简历），做出优先级
5. 联系优先级靠前的人
6. 同步用智谱 AutoClaw 再次验证结果

### 流程 B：新客户电话后自动生成客户文档

1. Codex 把全公司客户文档读一遍，输出客户文档的最佳标准并自调优
2. 每次和客户第一次电话后，把录音 + 公司介绍 + JD 等信息扔给 Codex
3. Codex 自动生成飞书文档版的客户文档

当前至少 6-7 步。未来愿景：打完电话后，文档自动生成，Agent 自动找人、推人，可能还没离开客户公司，候选人已经推出来了。

## Sourcing 的两大场景

### 场景 1：新客户快速嘎嘎推人
- 把库里存量合适的人迅速扫完
- 把 Boss 直聘上积极看机会的人迅速扫完
- 目标：把时间和成本降到最低
- 目前扫完需要好几周

### 场景 2：冰山下的人
- 水面上的扫完后
- 通过转介绍、Mapping、深挖等能力找冰山下的人
- 这是长期关键能力

## 老板的战略判断

1. **窗口期策略**：AI 时代先把场景 1 的效率和成本做到极致，在大家还没缓过来时做到几倍人效
2. **窗口期有限**：一旦成本被打下来，这块就赚不到钱了
3. **终局看场景 2**：真正的壁垒是找冰山下的人的能力
